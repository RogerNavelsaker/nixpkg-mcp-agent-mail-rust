# MODE_OUTPUT_L5: Scope Control Analysis

## Thesis

Asupersync has metastasized from a coherent "spec-first, cancel-correct async runtime" into a monolithic 708K-line crate (plus 282K lines of tests and 3.1 MB of documentation across 282 files) that attempts to simultaneously be a Tokio replacement, a web framework, a gRPC stack, three database clients, a brokerless messaging fabric, a fountain coding library, a formal methods workbench, a CLI diagnostic suite, and a WASM browser runtime. The core innovation -- structured concurrency with cancel-correctness -- is genuinely valuable and accounts for roughly 130K lines. The remaining 580K lines represent scope creep of varying severity, from defensible (networking, channels) to questionable (messaging fabric at 76K lines, RaptorQ at 26K) to actively harmful to sustainability (97 Tokio-replacement planning documents, a 22K-line CLI doctor). With one human maintainer and AI agents doing the bulk of implementation and auditing, the project has crossed the threshold where breadth undermines depth: no single person can hold the correctness invariants of 42+ modules in their head, and the documentation corpus has become a maintenance burden that outpaces the code it describes.

---

## Top Findings

### F1: Messaging Module Is a Second Project Hiding Inside the First

**Evidence**: `src/messaging/` is 75,765 lines across 25+ files. `fabric.rs` alone is 10,937 lines. The module includes a brokerless subject fabric, federation, IR compiler, morphisms, privacy framework, policy engine, consumer groups, and snapshot system -- on top of Redis, NATS, Kafka, and JetStream clients.

**Reasoning**: This is not "messaging client integration." This is a standalone distributed messaging system comparable in ambition to NATS or Pulsar. It imports from RaptorQ, distributed hashing, and the obligation system. The 76K lines exceed the entire runtime (61K) and represent a project that should be its own crate with its own release cycle.

**Severity**: CRITICAL  
**Confidence**: 0.95

---

### F2: Documentation Has Become Bureaucratic Overhead

**Evidence**: 282 documentation files totaling 3.1 MB. 97 files are Tokio-replacement planning documents alone. There are 40+ "semantic_" prefixed analysis documents, 30+ WASM-related docs, 25+ doctor-related contracts, and multiple "contract" documents that describe processes rather than code behavior.

**Reasoning**: Documentation-to-code ratio is pathological. The `docs/` directory contains more words than many entire codebases. Most of these documents were generated by AI agents and describe aspirational processes, planning matrices, and governance workflows that have no enforcement mechanism. Documents like `semantic_pareto_portfolio.md`, `semantic_profile_selection.md`, and `semantic_optimality_model.md` read as decision-theory exercises rather than engineering artifacts. The 97 Tokio replacement documents alone likely took more agent-hours to produce than the actual replacement code.

**Severity**: HIGH  
**Confidence**: 0.92

---

### F3: Obligation Module Contains a Research-Grade Formal Methods System

**Evidence**: `src/obligation/` is 30,813 lines including `separation_logic.rs` (2,429L), `no_aliasing_proof.rs` (2,140L), `no_leak_proof.rs` (1,475L), `dialectica.rs` (1,430L -- Dialectica interpretation), `lyapunov.rs` (2,300L+ -- Lyapunov stability certificates), `session_types.rs` (1,476L), `graded.rs` (1,351L -- graded monads), and a `choreography/` subdirectory with choreographic programming pipeline and codegen.

**Reasoning**: The core obligation concept (linear tokens that must be resolved before region close) is directly valuable and could be ~3-5K lines. But the module has expanded to include separation logic provers, Dialectica interpretations, choreographic programming, and graded monad tracking. These are PhD-level formal methods concepts that each deserve their own paper, let alone their own crate. The question is not whether they are intellectually interesting (they are), but whether they buy concrete correctness that simpler approaches cannot. The answer for most of these (Dialectica, choreography codegen, graded monads) is no -- they provide theoretical elegance for a runtime that already enforces obligations through simpler mechanisms.

**Severity**: HIGH  
**Confidence**: 0.85

---

### F4: RaptorQ Is a Standalone Library Embedded in a Runtime

**Evidence**: `src/raptorq/` is 25,554 lines implementing RFC 6330 systematic RaptorQ codes, including GF(256) arithmetic with SIMD (5,956L in `gf256.rs`), a full decoder (4,937L), linear algebra over GF(256), and a wavefront pipeline. The README justifies this as "distributed snapshot distribution" support.

**Reasoning**: RaptorQ fountain codes are a specialized erasure coding technique. This is a complete, optimized implementation that belongs on crates.io as its own crate. It has its own performance regression suite, its own test log schema, and its own decision contracts. The connection to the runtime (encoding region snapshots for distributed recovery) is a use case, not a justification for inclusion in the core runtime crate. There are existing RaptorQ crates; even if this implementation is better, it should be factored out.

**Severity**: MEDIUM  
**Confidence**: 0.93

---

### F5: Spectral Health Monitor Is Over-Engineered for the Problem

**Evidence**: `src/observability/spectral_health.rs` is 3,371 lines implementing Fiedler value tracking, Cheeger bounds, conformal prediction, e-processes, autocorrelation, variance ratio, flicker analysis, skewness, Kendall tau, Spearman rho, Hoeffding's D, and distance correlation -- all for wait-graph monitoring.

**Reasoning**: The stated goal is "detect structural degradation before deadlock." This is a valid goal. But the implementation deploys 12+ statistical indicators with anytime-valid testing, conformal calibration, and spectral graph theory where a much simpler approach (cycle detection + fan-out tracking + timeout escalation) would achieve 95% of the benefit at 5% of the complexity. The spectral approach requires computing eigenvalues of the wait-graph Laplacian, which is O(n^3) in the number of tasks. This is the quintessential "alien artifact" pattern: mathematically elegant, practically over-specified.

**Severity**: MEDIUM  
**Confidence**: 0.80

---

### F6: The CLI Doctor Module Is Disproportionately Large

**Evidence**: `src/cli/doctor/mod.rs` is a single 21,756-line file. The entire CLI module is 25,982 lines.

**Reasoning**: A diagnostic tool for an async runtime should not be larger than the runtime's scheduler (6,790 lines for `three_lane.rs`), all channels combined (11,957 lines), or all sync primitives combined (13,794 lines). The doctor module appears to contain scenario running, workspace scanning, evidence ingestion, report generation, and visual diagnostics -- each of which should be its own module, and most of which could be external tooling rather than compiled into the runtime library.

**Severity**: MEDIUM  
**Confidence**: 0.90

---

### F7: The Project Is Trying to Replace the Entire Tokio Ecosystem in One Crate

**Evidence**: The README contains a "Tokio Ecosystem Coverage Map" table that maps Tokio, Hyper, Axum, Tower, Tonic, tokio-postgres, mysql_async, tokio-tungstenite, trust-dns, and more to Asupersync equivalents. The project includes all of: runtime, channels, sync primitives, I/O, codecs, bytes, TCP/UDP/Unix, DNS, TLS, WebSocket, HTTP/1.1, HTTP/2, QUIC/HTTP/3, web framework, gRPC, database clients (Postgres/MySQL/SQLite), messaging (Redis/NATS/Kafka), streams, service middleware, process management, signals, observability, and tracing compatibility.

**Reasoning**: The Tokio ecosystem is maintained by dozens of people across many crates. Replacing it monolithically with one crate and one human maintainer (plus AI agents) creates a bus factor of 1 and means every module competes for the same maintenance bandwidth. The single-crate approach means a bug in the Kafka client blocks a release of the runtime. Feature-gating helps but does not solve the fundamental organizational problem.

**Severity**: HIGH  
**Confidence**: 0.95

---

### F8: Test Code Volume Approaches Source Code Volume

**Evidence**: 282,270 lines of test code in `/tests/` plus test modules embedded in source files. Combined with 708,258 lines of source, tests represent roughly 40% of total Rust code.

**Reasoning**: High test-to-code ratio is generally positive, but at this scale it creates its own maintenance burden. Every refactor to a core type ripples through hundreds of thousands of lines of tests. The AI agent audit notes in MEMORY.md document sessions auditing 100+ files at a time, suggesting the test corpus is maintained primarily by AI agents whose false positive rate is acknowledged ("Explore agent unreliable... ALWAYS verify audit findings manually").

**Severity**: LOW  
**Confidence**: 0.75

---

### F9: Formal Proofs Are Shallow Scaffolds, Not Deep Verification

**Evidence**: `formal/lean/Asupersync.lean` is 5,210 lines. `formal/tla/Asupersync.tla` is 792 lines. MEMORY.md claims "ALL 6 CORE INVARIANTS FULLY_PROVEN. 170 theorems."

**Reasoning**: 5K lines of Lean for a 708K-line runtime is a sketch, not a verified system. For comparison, seL4's full functional correctness proof is ~200K lines of Isabelle for ~10K lines of C. The Lean file likely proves properties of idealized models rather than the actual implementation. This is still valuable for design assurance, but the project should not claim formal verification -- it has formal modeling.

**Severity**: LOW  
**Confidence**: 0.85

---

### F10: Plan Module Contains an Optimizer for a DSL Nobody Uses Yet

**Evidence**: `src/plan/` is 14,032 lines including `latency_algebra.rs` (2,689L -- a semiring-based latency cost model), `rewrite.rs` (1,707L -- a combinator DAG rewriter), `analysis.rs` (2,841L), and `certificate.rs` (2,003L).

**Reasoning**: This module implements an optimization pass for combinator pipelines based on algebraic rewrite rules. It is the kind of infrastructure that makes sense when thousands of users are composing combinators and need automatic optimization. For a pre-1.0 runtime with no external users, this is premature optimization of the development platform itself. The 14K lines would be better spent hardening the protocol implementations.

**Severity**: MEDIUM  
**Confidence**: 0.82

## Risks Identified

1. **Bus factor = 1**: One human maintainer. AI agents can audit and implement but cannot own architectural decisions, respond to security disclosures, or engage with community feedback. If the maintainer becomes unavailable, 708K lines of Rust become unmaintainable.

2. **Monolith lock-in**: Everything in one crate means users cannot adopt just the runtime without compiling database clients, messaging fabric, and fountain codes. Feature gates mitigate compilation cost but not conceptual complexity.

3. **Documentation rot**: 3.1 MB of docs with no clear lifecycle policy. Many documents reference plans that are now complete, creating confusion about what is aspirational vs. actual.

4. **Agent-generated code trust**: The audit notes acknowledge ~65 bugs found across 587 files. While impressive audit coverage, the code was also largely agent-generated, creating a cycle where agents write code that other agents audit. The false positive rate for agent audits is noted as problematic.

5. **Complexity cliff for contributors**: Anyone trying to contribute must understand structured concurrency, capability security, obligation systems, formal semantics, spectral graph theory, fountain codes, choreographic programming, and Dialectica interpretations. The barrier to contribution is a PhD in each.

6. **Compile time**: 708K lines of Rust in one crate will have multi-minute compile times even on fast hardware. This directly impedes iteration speed.

## Recommendations

### P0 -- Do Immediately

1. **Extract RaptorQ into its own crate.** It has no API coupling to the runtime beyond being imported by the messaging module. Publish to crates.io. This removes 26K lines from the main crate.

2. **Extract the messaging fabric into its own crate.** `messaging/fabric.rs` and its 15+ supporting modules (ir, compiler, morphism, federation, etc.) are a standalone system. The Redis/NATS/Kafka clients can remain as thin integration modules.

3. **Archive or delete the 97 Tokio replacement planning documents.** Move them to a `docs/archive/tokio-planning/` directory with a note that the planning phase is complete. They are creating a false impression of ongoing planning work.

### P1 -- Do Soon

4. **Split the obligation module.** Keep core obligation tracking (~3-5K lines: ledger, mod, recovery, eprocess). Move separation logic, Dialectica, choreography, graded monads, and proofs into a `formal-methods` crate or feature-gated module that is not compiled by default.

5. **Split the CLI doctor into its own binary crate.** 22K lines of diagnostic tooling should not be compiled into the library.

6. **Extract database clients into separate crates.** `database/postgres.rs`, `database/mysql.rs`, and `database/sqlite.rs` implement full wire protocols. These should be `asupersync-postgres`, `asupersync-mysql`, `asupersync-sqlite`.

### P2 -- Do When Possible

7. **Simplify spectral health monitoring.** Replace or supplement the spectral approach with simpler cycle detection and timeout-based liveness checks. Keep the spectral monitor as an optional diagnostic, not a core path.

8. **Reduce the plan/rewrite optimizer to a sketch.** Flag it as experimental. Users are not composing combinator DAGs at scale yet.

9. **Establish a documentation lifecycle.** Tag each doc with a status (draft, active, archived). Auto-archive docs older than 6 months that have not been updated.

### P3 -- Consider

10. **Factor the web framework into its own crate.** `src/web/` at 15K lines is an Axum competitor. It could be `asupersync-web`.

11. **Factor gRPC into its own crate.** `src/grpc/` at 10K lines is a Tonic competitor.

12. **Factor HTTP into its own crate.** `src/http/` at 32K lines is a Hyper competitor. This is harder due to tighter runtime integration.

### P4 -- Long-term

13. **Consider whether the project should be a workspace of 8-12 crates rather than 1 monolith.** Core runtime, channels/sync, networking, HTTP, web, gRPC, database, messaging, lab, formal, and CLI as separate crates with independent versioning.

14. **Evaluate whether a second human maintainer is needed before any public release.** The bus factor risk is existential.

## New Ideas and Extensions

- A `cargo asupersync doctor` binary crate would let the diagnostic tooling evolve independently and ship as a separate install.
- An `asupersync-ecosystem` meta-crate that re-exports the workspace members would give users the monolith experience while enabling granular dependency selection.
- A "minimum viable Asupersync" feature profile that compiles only runtime + channels + sync + time + I/O would let users adopt the core innovation without the ecosystem weight.
- The RaptorQ crate, if published standalone, could attract its own community and contributors independent of the runtime.

## Questions for Project Owner

1. **What is the intended user?** Is this for the maintainer's own projects, for a specific company/team, or for the general Rust community? The answer dramatically changes scope priorities.

2. **Is the messaging fabric in production anywhere?** 76K lines of brokerless messaging is either the project's secret weapon or its biggest liability.

3. **What is the plan for the 97 Tokio planning docs?** Are they being maintained, or are they historical artifacts?

4. **Has the Lean proof been run recently?** Does it still type-check against current semantics?

5. **What is the target for first external user adoption?** The answer determines whether P0 items need to happen this month or this quarter.

6. **Is the spectral health monitor actually used in any test or production scenario?** Or is it a capability demonstration?

7. **Would you accept a PR that splits the crate into a workspace of 8+ crates?** This is the single highest-leverage structural change.

## Points of Uncertainty

- **Test quality vs. quantity**: 282K test lines could be thorough coverage or could be largely redundant golden-path tests. Without running coverage analysis, I cannot distinguish.
- **Feature gate effectiveness**: The project uses feature gates extensively. If most of the exotic modules are behind non-default features, the compile-time and complexity arguments weaken (though the maintenance argument does not).
- **Agent code quality**: The 65 bugs found in audit across 587 files is a ~11% defect rate per file, which is actually reasonable for a project this size. The question is whether the remaining unaudited code has similar or worse rates.
- **Performance**: If the spectral monitor, Lyapunov governor, and e-process oracle actually provide measurably better production outcomes than simpler alternatives, the complexity is justified. I have not seen benchmark evidence either way.

## Agreements and Tensions with Other Perspectives

- **L1 (Correctness)**: Would likely agree the core runtime is sound but worry about the correctness implications of the enormous attack surface area. More code = more bugs, and the formal proofs cover only a fraction.
- **L2 (Architecture)**: Would likely praise the structured concurrency model but flag the monolith as an architectural anti-pattern. The region/obligation model is elegant; the monolithic delivery vehicle is not.
- **L3 (Performance)**: Would note that spectral eigenvalue computation in the observability hot path is O(n^3), and that compile time is itself a performance issue for development velocity.
- **L4 (Usability)**: Would flag that the API requires PhD-level understanding of too many domains. The `Cx` threading pattern is learnable; Dialectica interpretations are not.
- **L6 (Risk)**: Would amplify the bus factor concern and note that a single-maintainer 708K-line project with formal verification claims is a reputational risk if bugs are found in "proven" invariants.

## Confidence: 0.82

**Calibration note**: High confidence in the structural findings (monolith scope, doc bloat, messaging size) because these are directly measurable. Lower confidence in the value judgments about formal methods modules (F3, F5) because the maintainer may have specific use cases that make them essential. The core thesis -- that scope has exceeded what one person can maintain -- is supported by the raw numbers but could be partially mitigated by effective AI agent tooling, which is an evolving capability. I have not run the code or profiled it, so performance-related claims are inferred rather than measured.
